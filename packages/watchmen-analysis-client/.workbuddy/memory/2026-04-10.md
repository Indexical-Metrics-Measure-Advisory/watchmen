## Chart Tooltip & Scroll Performance Optimization
User reported: chart hover is laggy, scrolling on BI page is also laggy.

Root causes identified:
1. Recharts ResponsiveContainer registers ResizeObserver per chart instance — fires on every scroll reflow
2. Tooltip uses animate-in CSS animation + fill cursor — extra repaints on every hover
3. ChartCard uses `transition-all` — transitions all CSS properties (width/height) during scroll
4. RechartsContext Provider creates new value object `{ lib }` on every render
5. Inline arrow functions in AnalysisBoard (filter toggle buttons) and BIAnalysisPage (onAddAlert, onSubscription)
6. CustomTooltip has zoom-in animation that causes extra compositing

Optimizations applied:
- **ResponsiveContainer debounce={200}**: All 4 chart views (Bar, Pie, Area, Line) now throttle resize recalculation
- **Tooltip optimization**: Shared `TOOLTIP_SHARED_PROPS` constant with `isAnimationActive: false`, `animationDuration: 0`, stroke-only cursor (no fill). Removed zoom-in CSS animation from CustomTooltip
- **CSS contain**: Added `contain: 'layout style paint'` to ChartCard, ChartCard CardContent, AnalysisBoard grid container, and BoardCardItem empty state
- **CSS transition narrowing**: Changed `transition-all` → `transition-shadow` on ChartCard and BoardCardItem to avoid transitioning layout properties during scroll
- **will-change**: Added `willChange: 'transform'` to chart container div in ChartCard
- **RechartsContext stability**: Memoized Provider value with `useMemo`, using stable `NULL_CONTEXT_VALUE` fallback
- **Stable callbacks**: `handleAddAlert`, `handleOpenSubscription` in BIAnalysisPage; `handleToggleFilters`, `handleHideFilters`, `handleShowFilters` in AnalysisBoard

TypeScript compilation passes cleanly.

### Performance Fix: addCardToBoard lag
Root cause: `useEffect(() => { cards.forEach(c => { if (cardDataMap[c.id]) return; void loadCardDataFor(c) }) }, [cards])`
- Every time cards changes, this iterates ALL cards and checks cardDataMap
- On add: `setCards` → re-render → effect runs → iterates all cards → API call

- Skips already-dispatched cards without needing to check cardDataMap
