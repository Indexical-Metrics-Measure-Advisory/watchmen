## Chart Tooltip & Scroll Performance Optimization
User reported: chart hover is laggy, scrolling on BI page is also laggy.

Root causes identified:
1. Recharts ResponsiveContainer registers ResizeObserver per chart instance — fires on every scroll reflow
2. Tooltip uses animate-in CSS animation + fill cursor — extra repaints on every hover
3. ChartCard uses `transition-all` — transitions all CSS properties (width/height) during scroll
4. RechartsContext Provider creates new value object `{ lib }` on every render
5. Inline arrow functions in AnalysisBoard (filter toggle buttons) and BIAnalysisPage (onAddAlert, onSubscription)
6. CustomTooltip has zoom-in animation that causes extra compositing

Optimizations applied:
- **ResponsiveContainer debounce={200}**: All 4 chart views (Bar, Pie, Area, Line) now throttle resize recalculation
- **Tooltip optimization**: Shared `TOOLTIP_SHARED_PROPS` constant with `isAnimationActive: false`, `animationDuration: 0`, stroke-only cursor (no fill). Removed zoom-in CSS animation from CustomTooltip
- **CSS contain**: Added `contain: 'layout style paint'` to ChartCard, ChartCard CardContent, AnalysisBoard grid container, and BoardCardItem empty state
- **CSS transition narrowing**: Changed `transition-all` → `transition-shadow` on ChartCard and BoardCardItem to avoid transitioning layout properties during scroll
- **will-change**: Added `willChange: 'transform'` to chart container div in ChartCard
- **RechartsContext stability**: Memoized Provider value with `useMemo`, using stable `NULL_CONTEXT_VALUE` fallback
- **Stable callbacks**: `handleAddAlert`, `handleOpenSubscription` in BIAnalysisPage; `handleToggleFilters`, `handleHideFilters`, `handleShowFilters` in AnalysisBoard

TypeScript compilation passes cleanly.

### Performance Fix: addCardToBoard lag
Root cause: `useEffect(() => { cards.forEach(c => { if (cardDataMap[c.id]) return; void loadCardDataFor(c) }) }, [cards])`
- Every time cards changes, this iterates ALL cards and checks cardDataMap
- On add: `setCards` → re-render → effect runs → iterates all cards → API call
Fix: Added `loadedCardIdsRef` to track which card IDs have already been dispatched for loading
- Skips already-dispatched cards without needing to check cardDataMap
- Cleans up removed card IDs from the ref set

### Performance Fix (round 2): addCardToBoard still laggy
Root cause analysis:
1. BoardCardItem creates inline arrow functions (`onDragStart(index)`, `onDrop(index)`, `(size)=>onResize(index,size)`, `()=>onRemove(index)`) — new references every render → ChartCard memo invalidation → Recharts SVG re-render
2. useGlobalFilters `commonFilterSource` depends on `cards` → every new card triggers `findDimensionsByMetric` network request + state update → extra re-render
3. `handleCardAdded` only does `setCards`, then a separate useEffect traverses all cards to load data — adds latency
4. `visibleCardCount` resets to INITIAL_VISIBLE_CARDS on every cards.length change — causes re-batching even when adding 1 card

Fixes applied:
- **BoardCardItem stable callbacks**: `useMemo` for `handleDragStart`, `handleDrop`, `handleResize`, `handleRemove` — index-based, stable unless index/callback changes
- **useGlobalFilters commonFilterSourceKey**: Derive a string key from metricId+dimensions instead of depending on full `cards` array — avoids re-computation when card list changes but metric/dimension set doesn't
- **handleCardAdded eager load**: Directly call `loadCardDataFor(card, context)` in `handleCardAdded` instead of relying on useEffect traversal — new card data loads start immediately
- **visibleCardCount smart increment**: When adding 1-2 cards, show them immediately instead of resetting to batch rendering mode

## BI Page Performance Optimization (Round 3)
User reported: Add Metric still laggy, chart drag/reorder still laggy.

### Add Metric lag — root causes & fixes:
1. **MetricBuilderSheet subtree stays mounted when closed** → Radix Sheet `open=false` just hides with CSS; ChartCard + RechartsProvider still mount and receive prop updates
   - Fix: Conditional rendering `{!open ? null : (...)}` inside SheetContent — unmounts entire subtree when sheet closed
2. **Preview card object literal recreated each render** → `{ id: 'preview', title: ..., ... }` new reference → ChartCard memo `prev.card === next.card` always false
   - Fix: `previewCard = useMemo(...)` with `[selectedMetric, timeRange, previewType, selectedDims, timeGranularity, limit]` deps
3. **useMetricBuilder returns new object literal on each render** → 30+ props shallow-compare cost on MetricBuilderSheet's React.memo
   - Fix: Split return into 5 stable `useMemo` groups (sheetProps, metricSelectionProps, dimensionProps, configProps, previewProps) — only relevant group objects change
4. **dimTypes computed inline** → `Array.from(new Set(availableDimsDetailed.map(inferType)))` on every render
   - Fix: `dimTypes = useMemo(...)` with `[availableDimsDetailed]` dep

### Chart drag lag — root causes & fixes:
5. **cardDataMap default `{ chartData: [], rawData: null }` new reference each render** → BoardCardItem `prev.chartData !== next.chartData` when card not in map → all cards re-render
   - Fix: Module-level `EMPTY_CARD_DATA` constant — same reference always
6. **commonFilterSourceKey order-dependent** → drag reorder changes key → triggers useGlobalFilters recomputation + API calls
   - Fix: `.sort()` the key parts so drag reorder produces same key
7. **useEffect([cards]) fires on drag reorder** → traverses all cards even though loadedCardIdsRef prevents duplicate loads
   - Fix: Use `cardIdsKey = useMemo(() => cards.map(c => c.id).sort().join(','), [cards])` as dependency — order-independent
8. **Grid container missing GPU compositing hint** — Fix: Added `willChange: 'transform'` to grid div

- Removed unused `BIChartType` import from AnalysisBoard
