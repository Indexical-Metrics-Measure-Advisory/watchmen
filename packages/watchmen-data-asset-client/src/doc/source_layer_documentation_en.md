# Insurance Company Business Data Classification Documentation

## 1. Data Source Introduction

### 1.1 Data Source Overview
- **Data File**: `source_data/gemini domain layer.csv`
- **Data Content**: Data information from various business departments of a life insurance company
- **Data Scale**: Contains 2,390 records covering detailed information from over 50 business tables in the company
- **Data Description**: Each record includes table name, field name, data type, length, required status, business description, and other information

### 1.2 Data Standardization File
- **Standardization File**: `doc/odps_source_layer.yaml`
- **File Purpose**: Business data specifications organized according to international data product standards
- **Content Included**: Data product definitions, business table structures, field specifications, data relationship mappings, etc.

## 2. Business Domain Classification

We have categorized all company business data into 11 major categories based on actual business functions, with each category managing related business data:

### 2.1 Business Domain Overview

| Business Domain | Abbreviation | Number of Tables | Main Business Responsibilities |
|----------------|--------------|------------------|-------------------------------|
| Financial Collection & Payment Management | BCP | 12+ | Manages all money-related business: accounts, transfers, collections, payments, etc. |
| Claims Service Management | CLM | 8+ | Manages customer claims: claim applications, reviews, payments, medical expenses, etc. |
| Policy Business Management | PA | 15+ | Manages policy lifecycle: underwriting, changes, renewals, surrenders, etc. |
| Product Management | PRD | 4+ | Manages insurance products: product design, liability definitions, product launch/withdrawal, etc. |
| Customer Information Management | PTY | 6+ | Manages customer data: individual customers, corporate customers, contact information, etc. |
| Sales Channel Management | SC | 3+ | Manages sales teams: agents, branches, sales organizations, etc. |
| Basic Code Management | CD | 2+ | Manages system basic data: various code tables, dictionaries, etc. |
| Proposal Application Management | PROP | 2+ | Manages underwriting process: proposal applications, underwriting reviews, etc. |
| Underwriting Management | UW | 9+ | Manages complete underwriting process: risk assessment, underwriting decisions, condition setting, extra loading management, etc. |

## 4. Data Organization Rules

### 4.1 Table Name Organization Rules
- **Original Table Names**: Start with "DM_", followed by business domain abbreviation, ending with "_HIS"
- **Organized Table Names**: Changed to start with "src_", all lowercase letters, connected with underscores
- **Example**: `DM_BCP_ACCOUNT_AUTH_HIS` â†’ `src_bcp_account_auth_his`

### 4.2 Field Name Organization Rules
- **Unified Format**: All field names changed to lowercase letters, words connected with underscores
- **Standard Fields**: Each table includes unified management fields such as record ID, creation time, update time, etc.
- **Business Fields**: Retain original business meaning, only format unified

### 4.3 Data Type Description
- **Numeric Types**: Used to store amounts, quantities, numbers, and other numeric information
- **Text Types**: Used to store names, addresses, descriptions, and other text information
- **Date Types**: Used to store various date and time information
- **Boolean Types**: Used to store yes/no choice information

## 7. Field Usage Instructions

### 7.1 Standard Field Descriptions
Each table includes the following standard management fields:
- `ID_`: System internal sequence number, used to uniquely identify each record
- `SRC_ENTITY_KEY`: Source system entity unique key
- `SRC_START_TIME`: Historical data start time
- `SRC_END_TIME`: Historical data end time (9999-12-31 indicates currently valid data)
- `INSERT_TIME`: Record creation time
- `UPDATE_TIME`: Record last update time
- `INSERTED_BY`: Record creator
- `UPDATED_BY`: Record updater

### 7.2 Association Relationship Descriptions
- **Policy Related**: Link all policy-related tables through `POLICY_ID` and `POLICY_NO`
- **Customer Related**: Link various customer information tables through `PARTY_ID`
- **Product Related**: Link product definition tables through `PRODUCT_ID` and `PRODUCT_CODE`
- **Claims Related**: Link all claims-related tables through `CASE_ID` and `CASE_NO`
- **Agent Related**: Link sales channel tables through `AGENT_ID` and `AGENT_CODE`
- **Proposal Related**: Link proposal and underwriting tables through `PROPOSAL_ID` and `PROPOSAL_NO`
- **Fund Related**: Link ILP fund-related tables through `FUND_ID` and `FUND_CODE`
- **Account Related**: Link policy account tables through `PAC_ACCOUNT_ID` and `ILP_ACCOUNT_ID`

### 7.3 Data Type Descriptions
- **BIGINT**: Large integer, typically used for ID fields and large numeric values
- **DECIMAL**: Precise numeric, used for amounts, ratios, and other financial data requiring precision
- **VARCHAR**: Variable-length string, used for text information such as names, descriptions, codes
- **DATETIME**: Date and time, used for timestamps requiring both date and time
- **DATE**: Date only, used for birthdays, due dates, effective dates, etc.
- **CHAR**: Fixed-length string, used for codes and flags with fixed formats
- **FLOAT**: Floating-point number, used for rates, percentages, and other numeric calculations

### 7.4 Business Usage Suggestions
1. **Query Policy Information**: Use `POLICY_NO` as primary key, link policy, coverage, customer, and other related tables
2. **Analyze Claims Data**: Use `CASE_NO` as primary key, link claims cases, coverage, medical expenses, and other tables
3. **Customer Profile Analysis**: Use `PARTY_ID` as primary key, link personal information, address, bank account, and other tables
4. **Sales Performance Analysis**: Use `AGENT_CODE` as primary key, link agent information and policy sales data
5. **Financial Analysis**: Link accounts receivable/payable, collection, payment, and other finance-related tables for fund flow analysis
6. **Underwriting Analysis**: Use `PROPOSAL_NO` as primary key, link proposal, underwriting conditions, exclusions, and other tables
7. **Investment Analysis**: Use `FUND_CODE` as primary key, link fund information, prices, investments, and other ILP tables
8. **Account Management**: Use account IDs as primary keys, link account values, transactions, and other account management tables

## 8. Business Domain Table Relationship Descriptions

### 8.1 Internal Relationships within Business Domains

#### 8.1.1 Financial Collection & Payment Management (BCP) Internal Relationships
- **Core Relationship**: Business transactions (`DM_BCP_BIZ_TRANS_HIS`) serve as the central hub, connecting to accounts receivable/payable (`DM_BCP_ARAP_HIS`), collections (`DM_BCP_COLLECTION_HIS`), and payments (`DM_BCP_PAYMENT_HIS`)
- **Account Management**: Bank account authentication (`DM_BCP_ACCOUNT_AUTH_HIS`) links to bank transfers (`DM_BCP_BANK_TRANSFER_HIS`) and check business (`DM_BCP_CHEQUE_HIS`)
- **Suspense Processing**: Suspense account summary (`DM_BCP_SUSPENSE_HIS`) connects to suspense account details (`DM_BCP_SUSPENSE_DTL_HIS`) and offset details (`DM_BCP_OFFSET_DTL_HIS`)

#### 8.1.2 Claims Service Management (CLM) Internal Relationships
- **Core Relationship**: Claims cases (`DM_CLM_CLAIM_CASE_HIS`) serve as the central hub, connecting to claims coverage (`DM_CLM_CLAIM_COVERAGE_HIS`), medical expenses (`DM_CLM_MEDICAL_FEE_HIS`), and liability records (`DM_CLM_CLAIM_LIAB_HIS`)
- **Assessment Process**: Claims assessment (`DM_CLM_ASSESSMENT_HIS`) and case assessment (`DM_CLM_CASE_ASSESSMENT_HIS`) both link to claims cases for comprehensive evaluation

#### 8.1.3 Policy Business Management (PA) Internal Relationships
- **Core Relationship**: Policy basic information (`DM_PA_POLICY_HIS`) serves as the central hub, connecting to policy coverage (`DM_PA_POLICY_COVERAGE_HIS`), policy changes (`DM_PA_POLICY_CHANGE_HIS`), and policy roles (`DM_PA_POLICY_ROLE_HIS`)
- **Premium Management**: Coverage premium (`DM_PA_COVERAGE_PREM_HIS`) links to premium discounts (`DM_PA_PREM_DISCNT_HIS`) and extra loadings (`DM_PA_EXTRA_LOADING_HIS`)
- **Application Process**: Application coverage (`DM_PA_APPL_COVERAGE_HIS`) connects to campaign definitions (`DM_PA_CAMPAIGN_DEF_HIS`) and producer records (`DM_PA_PRODUCER_HIS`)

#### 8.1.4 Underwriting Management (UW) Internal Relationships
- **Core Relationship**: Underwriting conditions (`DM_UW_CONDITION_HIS`) serve as the central hub, connecting to endorsements (`DM_UW_ENDORSEMENT_HIS`), exclusions (`DM_UW_EXCLUSION_HIS`), and extra loadings (`DM_UW_EXTRA_LOADING_HIS`)
- **Party Information**: Underwriting insured (`DM_UW_INSURED_HIS`) and policyholder (`DM_UW_POLICY_HOLDER_HIS`) information link to medical information (`DM_UW_MEDICAL_INFO_HIS`)
- **Product Assessment**: Underwriting product (`DM_UW_PRODUCT_HIS`) information connects to all other underwriting tables for comprehensive risk assessment

#### 8.1.5 Investment-Linked Insurance (ILP) Internal Relationships
- **Core Relationship**: ILP fund information (`DM_ILP_FUND_HIS`) serves as the central hub, connecting to fund prices (`DM_ILP_FUND_PRICE_HIS`) and investment records (`DM_ILP_INVESTMENT_HIS`)
